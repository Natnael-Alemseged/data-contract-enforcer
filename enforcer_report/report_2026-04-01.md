### Violation 1: Week 2 — `rubric_id` is SHA256, not UUID

```
check_id:    week2-digital-courtroom-verdicts.rubric_id.uuid_pattern
status:      FAIL
severity:    CRITICAL
actual:      9 non-UUID values in sample
expected:    format: uuid (^[0-9a-f-]{36}$)
```

**Interpretation:** The Digital Courtroom system generates `rubric_id` by computing `sha256(rubric.json)`, which produces a 64-character hex string. The contract expected UUID-v4 (36 characters with hyphens). This is not a data error — it is a deliberate design choice in the Week 2 system that was never documented at the interface level. Without a contract, any downstream system assuming UUID-v4 would silently receive a longer string and potentially truncate it, causing join failures or data corruption. The contract makes this schema deviation explicit and forces a conscious decision: either update the contract to accept SHA256 hex strings, or mandate the Week 2 system emit a proper UUID.

### Violation 2: Week 5 — Business-Key Identifiers, not UUID-v4

```
check_id:    week5-axiom-ledger-events.aggregate_id.uuid_pattern  (×12 fields)
status:      FAIL
severity:    CRITICAL
actual:      95 non-UUID values (e.g. "loan-APEX-0001", "credit-APEX-0001")
expected:    format: uuid (^[0-9a-f-]{36}$)
```

**Interpretation:** The Axiom-Ledger event sourcing system uses human-readable business keys (`loan-APEX-0001`, `agent-credit_analysis-sess-...`) instead of opaque UUIDs. This is a valid domain-driven design choice for event streams — the stream ID encodes semantic meaning (loan type, agent role). However, the contract generator inferred UUID format based on the `_id` field suffix convention. This is exactly the kind of hidden assumption that contracts expose: the naming convention `*_id` does not guarantee UUID format. The 12 failures across `aggregate_id`, `aggregate_type`, and metadata fields reveal that the Week 5 schema uses a fundamentally different identifier strategy than Weeks 1–3. Any system joining across weeks on `*_id` fields would need to handle both formats.

### Injected Violation Test: Confidence Scale Change (×100)

To prove the enforcer catches real-world failure modes, we injected a known violation: multiplying all `extracted_facts.confidence` values by 100, simulating the silent scale change from float [0.0, 1.0] to integer [0, 100] that was documented as a real risk in DOMAIN_NOTES Q2.

```
check_id:    week3-document-refinery-extractions.extracted_facts.confidence.range
status:      FAIL
severity:    CRITICAL
actual:      min=75.1000, max=98.7000, mean=86.8176
expected:    min>=0.0, max<=1.0
records:     102 failing (all records affected)
```

The statistical drift check also fired independently:

```
check_id:    week3-document-refinery-extractions.extracted_facts.confidence.statistical_drift
status:      FAIL
severity:    HIGH
actual:      mean=86.8176 (baseline=0.8682, z=1223.44)
expected:    |z| <= 3.0 (baseline stddev=0.0703)
```

**Interpretation:** The enforcer caught the scale change through two independent mechanisms: (1) the hard range constraint `[0.0, 1.0]` caught it immediately as a CRITICAL violation affecting all 102 records, and (2) the statistical drift detector flagged it at **1,223 standard deviations** from baseline — an extreme z-score that is unmistakable. In a real production pipeline, this is the difference between a confidence value of `0.317` being interpreted as "31.7% confident" vs "31,700% confident." The contract would have caught this before the corrupted data propagated to downstream consumers like the courtroom evaluator or the axiom ledger decision engine.

### Bitol v3.0.0 Contract — Week 3 Extractions (excerpt)

```yaml
kind: DataContract
apiVersion: v3.0.0
id: week3-document-refinery-extractions
schema:
  doc_id:
    type: string
    required: true
    format: uuid
    pattern: ^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$
  extraction_model:
    type: string
    required: true
    enum:
      - claude-3-5-sonnet-20241022
  extracted_facts.confidence:
    type: number
    required: true
    minimum: 0.0
    maximum: 1.0
    description: >
      Confidence score. MUST remain float 0.0–1.0.
      BREAKING CHANGE if converted to integer 0–100.
  processing_time_ms:
    type: integer
    required: true
    x-observed:
      min: 995.0
      max: 3532.0
      mean: 1830.2941
      stddev: 619.3054
quality:
  type: SodaChecks
  specification:
    checks for week3_document_refinery_extractions:
      - "- row_count >= 1"
      - "- missing_count(doc_id) = 0"
      - "- min(extracted_facts_confidence) >= 0.0"
      - "- max(extracted_facts_confidence) <= 1.0"
```

**Clause types present:** required fields (not-null), type enforcement (string/number/integer), format constraints (uuid, date-time), regex patterns, enum values, numeric ranges with observed statistics, and SodaChecks quality block. This goes well beyond field listing — each clause is a machine-checkable rule.

## 6. Reflection: What I Discovered by Writing Contracts

### Discovery 1: Field naming conventions hide incompatible identity strategies

Before writing contracts, I assumed that any field ending in `_id` would be a UUID-v4. This seemed reasonable — Weeks 1 and 3 both use standard UUIDs for their identifiers. But when the contract generator profiled Week 5's `stream_id` and `aggregate_id` fields, they contained values like `loan-APEX-0001` and `agent-credit_analysis-sess-cre-1284d40d`. These are human-readable business keys, not opaque UUIDs. Similarly, Week 2's `rubric_id` turned out to be a SHA256 hash (64 hex characters), not a UUID (36 characters).

This was invisible before contracts. If I had built a cross-system join on `*_id` fields assuming UUID-v4, it would have silently failed — strings would be truncated, joins would return zero rows, or hash lookups would miss. The contract enforcer caught these three distinct identifier strategies (UUID-v4, SHA256 hex, business-key string) across what appeared to be a uniform `*_id` naming convention. The lesson: naming conventions are not schema contracts.

### Discovery 2: Confidence scores are a ticking time bomb without range contracts

The most dangerous failure mode I found was not a crash — it was a silent scale change. Week 3's Document Refinery outputs `extracted_facts.confidence` as a float in [0.0, 1.0]. But when I profiled the actual runtime data, I realized that a simple ingest bug (multiplying by 100 for display) would turn `0.317` into `31.7`, and nothing in the pipeline would catch it. The value is still a valid float. It still sorts correctly. Downstream consumers would just interpret it 100× too high.

I assumed that type checking alone (number vs string) would catch data quality issues. It doesn't. The confidence scale change passes every type check — it's still a number. Only a hard range contract (`minimum: 0.0, maximum: 1.0`) catches it. And the statistical drift detector (z-score against baseline) provides a second independent signal: when confidence suddenly has a mean of 86.8 instead of 0.87, the z-score of 1,223 is unmistakable. I now understand why range and drift checks are not optional extras — they are the only defense against silent semantic corruption.

### Discovery 3: Schema migration is the most dangerous operation, and it's invisible without snapshots

I assumed that once a schema is defined, it stays stable. But the Week 5 canonical migration (converting raw event-sourcing fields to a standard format) renamed 9 columns in a single operation. The SchemaEvolutionAnalyzer flagged this as BLOCK — and it was right. If a downstream consumer was reading `aggregate_id` and it silently became `stream_id`, every query would return nulls. No error, no crash, just empty results.

What surprised me was that this migration was intentional and well-meaning — we were standardizing field names. But from the contract's perspective, renaming a column is identical to deleting it and adding a new one: it's a breaking change. Without schema snapshots to diff, this rename would have been invisible at the interface level. The consumer would discover it only when their dashboard went blank. This taught me that schema evolution analysis is not about catching mistakes — it's about making intentional changes visible before they propagate.

### What I would do differently

If I were starting over, I would write contracts *before* building the pipeline, not after. The Week 5 identifier issue and the confidence scale risk were both knowable at design time. A contract-first approach would have forced each system to declare its output schema explicitly, making these incompatibilities visible on day one instead of week seven.
